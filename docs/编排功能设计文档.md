### 1.2 已确认的问题

#### 🔴 P0: 拖拽不生效
**现象**: 在 App 运行时，左侧标签无法拖拽到右侧画布，或拖入后卡片不显示。

**根因分析**（待运行时验证）:
1. Tauri WebView2 中 HTML5 Drag & Drop API 的行为可能与浏览器不一致
2. `dragstart` 中 `dataTransfer` 的 `effectAllowed = 'copy'` 可能在某些 WebView 版本下无效
3. 画布区域 CSS `pointer-events` 或事件冒泡问题

**修复方案**: 保留 HTML5 DnD 作为首选，同时实现基于鼠标事件的 fallback：
- `mousedown` 记录拖拽起点和 tabId
- `mousemove` 在画布上显示幽灵卡片跟随鼠标
- `mouseup` 在画布区域内则创建卡片
- 若 HTML5 DnD 正常工作，鼠标事件方案不会冲突（可通过标志位判断）

#### 🟡 P1: 画布卡片不可二次移动
**现象**: 卡片拖入画布后位置固定，无法重新调整布局。

**修复方案**: 为卡片添加内部拖拽逻辑（不依赖 HTML5 DnD）：
- 卡片 `mousedown` 开始拖拽，记录 `offsetX/Y`
- `window.mousemove` 更新卡片 `left/top`
- `window.mouseup` 结束拖拽

#### 🟡 P2: 快照缺少画布信息
**现象**: 快照只保存了每个 tab 的 permission/title/session_id，未保存卡片位置和连线关系。

**修复方案**: 扩展快照数据结构，见「三、数据模型设计」。

## 二、五大目标设计方案

### 2.1 目标1: 编排分页 — 拖拽 + 连线控制通信路径

#### 2.1.1 连线可视化

在画布上增加一个 SVG 层，绘制卡片之间的通信连线：

```
┌─────────────────────────────────────────────────────────────┐
│  终端标签页         │              画布                      │
│                    │  ┌─────────┐      ┌─────────┐         │
│  ○ 前端构建        │  │ Tab #3   │◄────►│ Tab #5   │         │
│  ○ 后端服务 已编排  │  │ 前端构建  │      │ 后端服务  │         │
│  ○ 测试运行 已编排  │  └─────────┘      └─────────┘         │
│                    │       ▲                  ▲              │
│                    │       └──────────────────┘              │
│                    │              连线层(SVG)                 │
└─────────────────────────────────────────────────────────────┘
```

**连线规则**:
- 连线表示双向通信许可（从 A 画线到 B，意味着 A 的 allowed_targets 包含 B，同时 B 的 allowed_targets 包含 A）
- 连线可由两种方式产生：
  1. **手动连接**: 在卡片上按住连接点（小圆点）拖动到另一张卡片，建立连线
  2. **自动生成**: 当用户在权限弹窗中勾选目标后，画布自动渲染对应连线
- 删除连线：点击连线选中后按 Delete，或在弹窗中取消勾选

**SVG 连线绘制**:
- 使用 `<svg>` 绝对定位覆盖画布，与卡片同层
- 连线起点/终点为卡片边缘中心点，通过 `getBoundingClientRect` 计算
- 支持选中状态（高亮）、悬停提示（显示通信方向）

#### 2.1.2 连线与权限的双向绑定

| 操作 | 权限数据变化 | 连线变化 |
|------|-------------|---------|
| 弹窗中勾选 Tab B | A.allowed_targets 增加 B | 自动渲染 A→B 连线 |
| 弹窗中取消勾选 Tab B | A.allowed_targets 移除 B | 自动移除 A→B 连线 |
| 画布上从 A 画线到 B | A.allowed_targets 增加 B, B.allowed_targets 增加 A | 渲染双向连线 |
| 画布上删除 A→B 连线 | A.allowed_targets 移除 B, B.allowed_targets 移除 A | 连线消失 |

> 注：连线的"双向"是视觉上的表现。底层权限模型仍然是单向的 `allowed_targets` 列表。连线存在 = 两个方向都允许。

#### 2.1.3 拖拽修复与增强

**左侧标签 → 画布（拖入）**:
- 方案A（保留）: HTML5 DnD `dragstart`/`drop`
- 方案B（Fallback）: 鼠标事件 `mousedown` + `mousemove` + `mouseup`，在画布上显示幽灵卡片
- 两张方案共存，优先使用 HTML5 DnD，若检测到不支持则自动降级

**画布内卡片移动（新增）**:
- 卡片 `mousedown` 开始内部拖拽
- `mousemove` 限制在画布边界内更新 `x/y`
- `mouseup` 结束，更新 `canvasItems`

**画布内卡片连线（新增）**:
- 每张卡片边缘设置"连接锚点"（4个小圆点，上下左右）
- 从锚点 `mousedown` 开始拉线，跟随鼠标显示临时 SVG 线
- `mouseup` 在另一张卡片上则建立连接；在空白处则取消

### 2.2 目标2: Agent 职责定义 + 系统提示词注入

#### 2.2.1 调研结论

经代码分析和网络调研，Claude Code CLI **不支持**通过环境变量注入系统提示词。Claude Code 的系统提示是内部硬编码的，没有公开的环境变量接口（如 `CLAUDE_CODE_SYSTEM_PROMPT`）。

因此采用 **C 方案：启动后自动输入**。

#### 2.2.2 方案设计

**核心思路**: Claude Code 启动后，以用户身份自动发送第一条"角色设定"消息，使其在对话历史中承担特定职责。

**数据模型**:

```typescript
interface AgentRole {
  name: string           // 角色名称，如"前端架构师"
  description: string    // 角色描述，如"负责组件设计和代码审查"
  systemPrompt: string   // 注入的系统提示词内容
}
```

**注入时机与流程**:

```
1. 用户为画布上的卡片配置 AgentRole
2. 终端标签启动时（pty_create），记录该 tab 需要注入的 systemPrompt
3. 监听 pty_title 事件，当该 tab 首次收到 OSC 0 标题时
   → 判定 Claude Code 已完成初始化
4. 延迟 500ms 后，通过 pty_write 写入 systemPrompt + \r\n
5. Claude Code 将其作为第一条用户消息处理
```

**注入内容格式**:

```
从现在开始，你的角色是：{role.name}。
{role.systemPrompt}
请始终以上述角色身份回应后续所有请求。
```

**注意事项**:
- 注入内容会出现在对话历史中，用户可见
- 若恢复 session（`-r session_id`），则**不重复注入**（session 已有历史上下文）
- 提供「重新注入」按钮，允许用户手动刷新角色设定

**存储位置**:
- AgentRole 定义存储在前端 `tabComm.ts` store 中，按 `tabId` 关联
- 持久化到快照中（`SnapshotTabEntry` 扩展 `role` 字段）
- 预设中作为 `VirtualAgent.role` 存储

#### 2.3.3 恢复流程（C方案：恢复 session）

```
用户选择加载快照
  │
  ▼
显示恢复确认弹窗（列出要恢复的 tabs 和 sessions）
  │
  ▼
按快照顺序创建终端标签
  ├─ 若 session_id 存在 → 启动时传入 -r session_id
  ├─ 若 session_id 不存在 → 全新启动
  └─ 应用该 tab 的 envVars、permission
  │
  ▼
等所有标签创建完成后
  ├─ 恢复画布卡片位置（canvas.items）
  ├─ 恢复连线（canvas.connections）
  ├─ 应用 AgentRole（新 session 则注入系统提示词）
  └─ 恢复 allowed_targets 权限
  │
  ▼
切换到编排分页，展示恢复后的拓扑
```

**关键问题**: 快照中的 `tab_id` 是历史值，恢复时新创建的 tab 会有新的 `tab_id`。需要在恢复时建立「旧 tab_id → 新 tab_id」的映射表，并据此重建连线关系和权限配置。

### 2.4 目标4: 编排预设（Preset）

#### 2.4.1 预设数据模型

```typescript
interface OrchestrationPreset {
  id: string                    // 唯一标识 (UUID)
  name: string                  // 显示名称，如"前后端协作模式"
  description?: string          // 预设描述
  createdAt: string             // ISO 时间
  updatedAt: string
  // 虚拟 Agent 定义（不绑定具体 tab，应用时才实例化）
  agents: VirtualAgent[]
  // 拓扑连线（基于 agent.id，不是 tabId）
  connections: VirtualConnection[]
  // 相对位置 (0-1)，便于适配不同画布尺寸
  layout: Record<string, { x: number; y: number }>
}

interface VirtualAgent {
  id: string
  name: string                  // 显示名称
  role: AgentRole
  launchConfig: {
    type: 'claude' | 'terminal'
    // 启动时覆盖的环境变量
    envOverrides: Record<string, string>
    // 其他启动参数（如 --dangerously-skip-permissions）
    args: string[]
  }
}

interface VirtualConnection {
  from: string   // VirtualAgent.id
  to: string     // VirtualAgent.id
}
```

#### 2.4.2 预设管理功能

**预设列表**（编排分页新增区域）:
- 在左侧边栏底部新增「预设管理」区域
- 显示已保存预设列表（名称 + 描述 + Agent 数量）
- 操作：应用 / 编辑 / 删除

**创建预设**（两种方式）:
1. **手动创建**: 点击「新建预设」，填写预设名称，逐个添加虚拟 Agent 和连线
2. **从当前编排创建**: 点击「保存为预设」，将当前画布上的拓扑（包括角色定义）保存为预设模板

**应用预设**:
```
选择预设
  │
  ▼
确认应用（显示要创建的 Agent 列表）
  │
  ▼
为每个 VirtualAgent 创建终端标签
  ├─ 启动对应 CLI（Claude Code / 普通终端）
  ├─ 传入 envOverrides 和 args
  ├─ 注入角色设定（新 session）
  └─ 在画布上创建卡片（使用 layout 的相对坐标换算为绝对坐标）
  │
  ▼
根据 VirtualConnection 建立权限和连线
  ├─ 设置每个 tab 的 allowed_targets
  └─ 渲染 SVG 连线
```

**预设持久化**:
- 存储路径: `%APPDATA%\ClaudeEnvManager\orchestration_presets\<preset-id>.json`
- Tauri 命令: `list_presets`, `save_preset`, `load_preset`, `delete_preset`

#### 2.4.3 Agent 自动创建预设（通过 tab-send）

支持从任意终端标签页通过 `tab-*` 命令创建预设：

```bash
# 创建一个预设（JSON 格式）
tab-preset-create '{"name":"代码审查流程","agents":[{"id":"reviewer","name":"代码审查员","role":{"name":"Senior Code Reviewer","systemPrompt":"你是一位资深代码审查专家..."},"launchConfig":{"type":"claude"}}],"connections":[{"from":"reviewer","to":"developer"}]}'

# 列出所有预设
tab-preset-list

# 应用预设（创建对应终端标签）
tab-preset-apply "preset-id"
```

**实现方式**: 在 `tab_cli.rs` 中新增 `Preset` 子命令，与现有命令解析和执行模式一致。


### 4.3 关键交互流程

#### 4.3.1 建立连线

```
用户在卡片A的连接锚点上 mousedown
  │
  ▼
OrchCanvas 进入"拉线模式"
  ├─ 显示临时 SVG 线，跟随鼠标
  └─ isDraggingConnection = true
  │
  ▼
鼠标移动到卡片B上，mouseup
  ├─ 建立 CanvasConnection { from: A.tabId, to: B.tabId }
  ├─ 双向设置权限:
  │   ├─ A.allowed_targets 增加 B.tabId
  │   └─ B.allowed_targets 增加 A.tabId
  ├─ 调用 set_tab_permission 保存
  └─ 渲染永久 SVG 连线
```

#### 4.3.2 系统提示词注入

```
终端标签创建（Claude Code 启动）
  │
  ▼
监听 pty_title 事件
  │
  ▼
首次收到该 tab 的 OSC 0 标题
  ├─ 判定 Claude Code 已就绪
  ├─ 检查该 tab 是否有配置 AgentRole
  ├─ 检查该 tab 是否为新 session（无 sessionId 或恢复失败）
  │
  ▼
延迟 500ms 后调用 pty_write(tabId, rolePrompt + "\r\n")
  │
  ▼
Claude Code 将角色设定作为首条用户消息处理
```

## 七、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| HTML5 DnD 在 WebView2 中完全不可用 | 高 | 已实现鼠标事件 fallback 方案 |
| 系统提示词自动注入时机不确定 | 中 | 通过 OSC 0 标题事件检测就绪状态，支持手动重新注入 |
| 快照恢复时 session 已过期 | 中 | 恢复失败时自动降级为全新启动，提示用户 |
| 画布上大量卡片和连线的性能 | 低 | SVG 使用虚拟渲染，仅渲染可视区域内的连线 |
| 多个 Agent 同目录下配置冲突 | 低 | 不依赖文件系统配置，所有配置通过 PTY 动态注入 |
