# 跨标签页 Agent 通信系统

## 设计目的

内置终端的多标签页天然隔离，A 标签页的 Agent 无法直接操作 B 标签页。为了让多个 Agent（如不同 Claude Code 会话）能够协同工作——比如一个 Agent 分析日志后让另一个 Agent 执行修复——需要在 PTY 输入层拦截特定命令，实现跨标签页的消息路由、输出读取和活跃检测。

随着标签页数量增加，直接在每个标签旁配置权限会变得杂乱。因此引入**编排**概念：将权限管理从终端页面剥离，集中到独立的编排视图中，以画布方式直观呈现多 Agent 之间的通信拓扑。

## 设计目标

1. **零侵入通信**：Agent 通过普通的终端输入 `tab-*` 命令即可与其他标签页交互，无需修改 Agent 本身
2. **权限可控**：每个标签页可独立启用/禁用通信功能，并可限制只允许与特定标签页通信
3. **可视化编排**：通过拖拽方式在画布上构建 Agent 通信拓扑，权限配置在画布卡片上完成
4. **会话可恢复**：支持按项目路径保存/加载终端快照（含权限配置），便于恢复多 Agent 工作流
5. **异步交互**：支持 `--wait` 模式的请求-响应机制，一个 Agent 发送任务后可挂起等待另一个 Agent 的回复

### 数据流：tab-send 命令

```
Agent (Tab 1) → 终端输入 "tab-send --to 2 "分析结果""
  → pty_write 拦截（检测 tab- 前缀）
    → tab_cli::parse_tab_command → TabCommand::Send
      → tab_cli::execute_command
        ├─ check_permission_inline(Tab 1, target=2) → 通过/拒绝
        ├─ 写入 Tab 2 的 PTY writer（Agent 2 在终端中看到消息）
        ├─ 写回 [TAB-CMD] ACK 到 Tab 1
        └─ （--wait 时注册 PendingReply，等待回复/超时）
```

### 核心数据结构

```rust
// 命令枚举
enum TabCommand {
    Send { to: u32, message: String, wait_seconds: Option<u64> },
    List,
    Read { from: u32, lines: usize },
    Presence { to: u32 },
}

// 权限配置（内联存储在 PtySession 中）
struct TabPermission {
    enabled: bool,
    allowed_targets: Vec<u32>,  // 空 = 允许所有
}

// --wait 挂起回复
struct PendingReply {
    msg_id: String,
    deadline: Instant,
    caller_tab: u32,
    responder_tab: u32,
    original_message: String,
}
```

### 命令拦截机制（pty_write）

`pty_write` 采用三阶段处理：

1. **早期放行**：缓冲区不以 `tab-` 开头 → 直接写入 PTY，零延迟
2. **行级解析**：以 `tab-` 开头 → 按 `\r`/`\n` 拆分，逐行识别完整命令
3. **命令分发**：识别为 `tab-*` → 调用 `execute_command`；非命令行 → 透传回 PTY

```rust
// --wait 超时清理也在 pty_write 入口执行
// 过期 PendingReply 会自动写入超时通知到原调用方 PTY
```

### 权限检查

```rust
fn check_permission_inline(session: &PtySession, target_id: Option<u32>) -> Result<(), String> {
    if !session.permission.enabled { return Err("Permission denied"); }
    if let Some(tid) = target_id {
        if !session.permission.allowed_targets.is_empty()
            && !session.permission.allowed_targets.contains(&tid) {
            return Err("Target not in allowed list");
        }
    }
    Ok(())
}
```

## 编排分页设计

### 设计动机

当标签页数量增多时，在每个标签旁单独配置权限会产生两个问题：
1. **认知负担**：用户需要在多个标签之间切换才能了解完整的通信拓扑
2. **操作分散**：权限配置入口分散在终端侧边栏的每个标签上，难以批量管理

编排分页将权限管理提升为独立的可视化操作，让用户以"画布"方式直观地看到哪些 Agent 参与了通信，以及它们之间的权限关系。

## CLI 命令集

Agent 在终端内输入以下命令即可与其他标签页通信：

| 命令 | 说明 |
|------|------|
| `tab-send --to <tab_id> "<message>" [--wait <seconds>]` | 向目标标签页发送消息，可选挂起等待回复 |
| `tab-list` | 列出所有标签页（ID, 状态） |
| `tab-read --from <tab_id> --lines <N>` | 读取目标标签页最近 N 行输出 |
| `tab-presence --to <tab_id>` | 检测目标标签页 Agent 是否活跃 |

### 使用示例

```bash
# 1. 发现可用标签
tab-list
# → [TAB-CMD] list: Tab 1 | Terminal | alive
#                  Tab 2 | Terminal | alive

# 2. 发送任务（即时）
tab-send --to 2 "请分析构建日志并汇报"
# → [TAB-CMD] send: Message sent to tab 2 (100 bytes)

# 3. 发送任务并等待回复
tab-send --to 2 "当前状态如何？" --wait 60
# → [TAB-CMD] send: Message sent to tab 2 (waiting for reply, timeout: 60s)
# （当 Tab 2 回复时）
# → [TAB-CMD] send: Reply received from tab 2: 构建成功

# 4. 读取对方输出
tab-read --from 2 --lines 10

# 5. 检测对方存活
tab-presence --to 2
```

## 权限模型

每个标签页有独立的权限配置：

```
TabPermission {
    enabled: bool,           // 是否启用跨标签通信
    allowed_targets: [u32]   // 允许通信的目标标签 ID 列表
}
```

- `enabled = false`：该标签的 Agent 无法发起任何 tab-* 命令
- `enabled = true, allowed_targets = []`：允许与所有标签通信
- `enabled = true, allowed_targets = [2, 5]`：只允许与 Tab 2 和 Tab 5 通信

权限检查发生在 `execute_command` 的入口，**在消息写入目标 PTY 之前**。如果权限不足，调用方会立即收到 `[TAB-CMD] Error: Permission denied`。

## 项目扫描报告 (2026-05-26)

由子 Agent 全量扫描代码后发现的实现缺口：

### 🔴 CRITICAL — `--wait` 模式完全不可用

**Bug 1: `pending_actions` 从未被应用**
`pty/mod.rs:299-316` 调用 `execute_command` 后只读取 `result.immediate_output`，完全忽略 `result.pending_actions`。这导致 `--wait` 模式永远不会注册 `PendingReply`，等待机制形同虚设。

**Bug 2: `check_and_fire_pending_replies` 搜索了错误的 map key**
`tab_cli.rs:390-416` 搜索 `pending_replies.get_mut(&target_id)`，但条目实际存储在 `pending_replies[caller_id]`（等待回复的那一方）。方向完全反了。

### 🟠 HIGH

- **`tab-list` 标题硬编码为 "Terminal"** — `PtySession` 没有 `title` 字段，OSC 0 标题提取后没有存回 session
- **`session_id` 快照中始终为 `None`** — 前端已解析，但 Rust 端 `pty_create` 从未接收该参数

### 🟡 MEDIUM

- 快照只恢复权限，不恢复终端内容（设计目标 4 的"会话恢复"打了折扣）
